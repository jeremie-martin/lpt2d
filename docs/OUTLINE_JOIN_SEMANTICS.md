# Outline Join Semantics

This document defines the conceptual model for hard, smooth, and rounded
boundary joins in lpt2d. It is a design reference, not an implementation
plan. Use it to evaluate whether future renderer, GUI, Python, JSON, and
authoring changes are moving the engine in the intended direction.

## Why This Exists

lpt2d needs to support authored 2D solid shapes that mix:

- straight runs
- visually smooth runs
- intentionally hard joins
- intentionally rounded joins

Today, polygon smooth shading is useful but too tightly coupled to polygon
topology. The thick-arc example makes that mismatch obvious: a segmented inner
curve is concave topologically, but is still intended to behave as one smooth
surface.

The engine needs a model centered on authored surface intent rather than on
`convex` vs `concave` as the primary concept.

## Current State

Today, `Polygon` exposes two different classes of controls:

- `smooth_angle`
  Shading-only normal interpolation on eligible polygon joins.
- `corner_radius` / `corner_radii`
  Real rounded boundary geometry.

The current smooth-shading behavior is intentionally conservative:

- smoothing is enabled only when `smooth_angle > 0`
- only convex zero-radius polygon joins are eligible
- concave joins stay flat
- beveled joins stay flat at the arc-adjacent ends

This is defensible as a safety heuristic, especially in a path tracer where the
shading normal affects bounce direction. But it is not the right long-term
model, because the heuristic is answering the wrong question. It is asking
"is this join convex?" when the engine really needs to answer "is this join
intended to be hard, smooth, or rounded?"

## Problem Statement

One polygon currently carries too many jobs at once:

- literal hit geometry
- approximation of some smoother intended form
- shading continuity carrier

That makes simple cases work and ambiguous cases feel inconsistent.

The design target is not "make concave polygon smoothing work somehow." The
design target is:

- author arbitrary 2D solid outlines cleanly
- keep hard, smooth, and rounded behavior conceptually distinct
- avoid faceting artifacts where a boundary is intended to read as one
  continuous surface
- preserve physically legible behavior instead of hiding geometry changes

## Scope

This document covers:

- terminology for shape boundary semantics
- the distinction between shading continuity and geometric continuity
- default behavior expectations
- the intended long-term outline model for lpt2d

This document does not cover:

- storage layout or exact data structures
- GUI control layout
- JSON schema details
- migration steps
- implementation ordering

## Core Distinctions

### Boundary Geometry

The boundary geometry is the actual shape that the ray intersects.

Examples:

- a polygon edge
- an arc
- a bezier segment
- a rounded corner arc generated by a fillet

Changing boundary geometry changes the physical shape.

### Shading Normal

The shading normal is the normal used for reflection and refraction at the hit
point.

In lpt2d this is not just a viewport cosmetic. The shading normal changes the
light transport result, so any smoothing rule must be explicit and defensible.

### Join Intent

A join is the place where two adjacent boundary segments meet.

For polygons, a join is a polygon vertex with one incoming edge and one
outgoing edge. For future mixed outlines, a join is the node between adjacent
line, arc, or bezier segments.

Join intent is the authored answer to:

- should the normal jump here?
- should the normal continue smoothly here?
- should this corner be replaced by real rounded geometry?

This is the primary concept. Convexity is not.

## Glossary For This Model

### Segment

A single boundary piece. Today this may be a polygon edge, arc, or bezier.
Longer-term it is one member of a closed outline.

### Join

The authored boundary connection between two adjacent segments.

### Sharp Join

A join with a shading discontinuity. The normal jumps across the join.

This is how a physically sharp edge remains optically sharp even when other
parts of the same shape are smooth.

### Smooth Join

A join with shading continuity. The boundary may still be piecewise linear, but
optically it should behave like one continuous surface.

This is a shading rule, not a geometry rule.

### Rounded Join

A join replaced by real geometry, such as a fillet arc. This changes the
boundary itself.

This is a geometry rule, not merely a shading rule.

### Auto Smooth

A rule that infers whether a join should be sharp or smooth from local
geometric information such as angle. Auto smooth is a default inference
mechanism, not the source of truth for authored intent.

## Decisions

The following points are the intended design direction for lpt2d.

### 1. Hard Edges Are A Real Authored Concept

Hardness is not merely a geometry trick or an artifact of low tessellation.
Some joins are intended to remain optically sharp even when the rest of the
shape is smooth.

### 2. Smooth Shading And Rounded Geometry Are Separate Concepts

These must remain distinct:

- `smooth`
  shading continuity only
- `rounded`
  geometric continuity

A shape can be:

- smooth but not rounded
- rounded but not globally smooth
- partly smooth, partly sharp, with some rounded joins

### 3. Join Intent Is More Fundamental Than Convexity

`convex` and `concave` are useful geometric facts, but they are not the
authored semantic model.

Convexity may still be used as a conservative heuristic in `auto` mode, but it
must not be treated as the primary meaning of a join.

### 4. Polygons Remain Literal Geometry

Polygon smooth shading may approximate a smoother intended surface, but it does
not silently turn polygon edges into curved geometry.

If the engine wants real curved geometry, it should represent real curved
geometry.

### 5. Rounded Joins Should Continue To Be Real Geometry

The existing rounded-corner model is valuable because it changes the actual
boundary shape instead of faking the result through normals alone.

That distinction should be preserved.

### 6. The Long-Term Primary Model Is A Closed Outline With Typed Segments

The engine should eventually support closed outlines built from a mix of:

- line segments
- arc segments
- bezier segments

with join semantics that stay meaningful across all of them.

In that model, a polygon is simply the special case where every segment is a
line segment.

## Desired Join Modes

For shading continuity, the intended conceptual join modes are:

- `auto`
- `sharp`
- `smooth`

`rounded` is not another shading mode. It is a separate geometric operation
that may coexist with the shading model.

The intended meaning is:

- `auto`
  infer smooth vs sharp from geometry, typically angle-based
- `sharp`
  force a normal discontinuity
- `smooth`
  force shading continuity

Explicit join intent should override auto inference.

## Expected Behavior

The following examples define the intended semantics.

### Square

A square should stay sharp by default. Enabling smooth shading must not be
confused with adding rounded corners.

If auto smoothing is angle-based, a square should remain sharp unless the user
explicitly overrides a join or changes the geometry.

### Thick `C`

A segmented thick arc that reads as a `C` should behave like this:

- both curved chains may smooth under `auto` if they are clearly approximating
  one continuous curved run
- the cap joins should stay sharp unless explicitly rounded
- if the cap geometry is actually rounded, then it should be rounded because of
  geometry, not because smooth shading invented the result

### Star

A star should not automatically become soft everywhere merely because global
smooth shading is enabled. Inward notches and spike joins are strong candidates
to remain sharp under `auto`.

### Sword / Knife Outline

A sword or knife should be authorable as one solid outline with:

- some runs intended smooth
- some joins intended hard
- some joins optionally rounded

The author should not need to fight polygon topology to express that.

### Serrated Blade

The long body of the blade may be smooth while the serrations stay sharp.
Auto smoothing should prefer this interpretation, and explicit overrides should
allow the author to lock the intended result.

## Design Principles For Future Work

Any future implementation should preserve the following principles.

### Author Intent Over Incidental Tessellation

The engine should follow what the outline is meant to represent, not merely how
many short segments happened to be used to approximate it.

### Safe Defaults

Auto behavior should be conservative where intent is ambiguous. Explicit
overrides exist for the cases where the engine cannot infer intent safely.

### No Silent Geometry Fiction

Shading rules may change normals. They should not silently claim that a shape
has rounded geometry when it does not.

### One Shared Story Across Surfaces

The same concepts must make sense in:

- C++ core
- Python authoring
- JSON
- GUI
- GPU path

### Future Compatibility

Anything added for polygons should extend naturally to future mixed
line/arc/bezier closed outlines rather than becoming polygon-only special
logic.

## From Today To The Target Model

Today:

- polygon smoothing is a convexity-gated heuristic
- polygon rounding is a real geometric operation
- there is no first-class authored join semantics model yet

Target direction:

- join intent becomes the conceptual foundation
- `auto | sharp | smooth` expresses shading continuity
- `rounded` remains a separate geometric boundary operation
- polygons keep working as literal line-segment outlines
- future closed mixed outlines inherit the same join semantics

## Questions This Document Should Answer Later

When evaluating future changes, use this document to ask:

- does this change preserve the distinction between smooth, sharp, and rounded?
- is the primary authored concept the join, or is the change still tied too
  tightly to convexity?
- is this a shading change, a geometry change, or both?
- are defaults conservative when intent is ambiguous?
- does this scale to future mixed line/arc/bezier outlines?

If the answer to those questions is consistently "yes", then the engine is
moving in the right direction.
